Engines & Models

T5 Tag-to-Text Summaries

Tags tell you what is on the board; they don't read like a human talking about it. The T5 path closes that last gap. Give it a position or a whole puzzle line, and crtk pulls tags — material balance, king safety, the tactical motif that matters — from the one shared chess core, wraps them in a fixed prompt, and runs a sequence-to-sequence T5 model end to end to return two to four sentences of commentary. The whole runtime is self-contained, dependency-free, and runs in-process. Decoding runs on the CPU, with optional CUDA / ROCm / oneAPI backends that fall back to CPU automatically. It works, but it is a research feature — read Status and quality before you trust the prose.

What it does

  • Converts deterministic position and puzzle-line tags into natural-language commentary.
  • Runs two CLI commands: fen text for single positions or FEN files, and puzzle text for expanded puzzle PVs.
  • Loads a custom .bin T5 export (weights, config, and a SentencePiece vocabulary in one file).
  • Tokenizes with a built-in SentencePiece unigram tokenizer — no Python, no native ML runtime required.
  • Generates greedily on the CPU, or on an optional GPU backend with transparent CPU fallback.

The tags driving the model come from the same code behind fen tags and puzzle tags. Nothing is invented along the way — the text describes precisely the motifs crtk detected, no more. See Tag reference for the vocabulary.

Pipeline at a glance

Stage What happens Source
Tag extraction Deterministic position/line tags are produced from the shared chess core fen tags, puzzle tags
Optional analysis Engine analysis enriches tags with eval and PV motifs --analyze
Prompt building Tags are formatted into a fixed instruction prompt TagPrompt
Tokenization Prompt is encoded with a SentencePiece unigram model Tokenizer
Generation T5 encoder-decoder runs greedy decoding to EOS or --max-new Runner
Decoding Output token ids are detokenized back to text Tokenizer

Quick start

Summarize one position:

crtk fen text --fen "r1bqkbnr/pppp1ppp/2n5/4p3/2B1P3/5Q2/PPPP1PPP/RNB1K1NR w KQkq - 4 4" --model models/t5.bin

Emit machine-readable output (one JSON object with fen and summary):

crtk fen text --fen "<FEN>" --model models/t5.bin --include-fen

Summarize a batch from a FEN file:

crtk fen text -i seeds.txt --model models/t5.bin --include-fen

The model path defaults to t5-model-path in config/cli.config.toml (shipped value: models/t5.bin). The repository ships no weights, so --model has to point at a .bin export you bring yourself. See Status and quality.

Position summaries: fen text

fen text tags a single position — or every FEN in a file — and feeds the result through the summarizer. Reach for it when you want commentary on one position and nothing more.

Option Meaning
--model PATH T5 .bin model path (default: config t5-model-path)
--fen FEN Input FEN (required unless --input is set)
--input / -i PATH Input FEN file
--include-fen Emit JSON with fen + summary
--max-new N Max generated tokens (default: 128)
--analyze Run engine analysis to enrich tags
--protocol-path / -P PATH Engine protocol TOML file
--max-nodes N Max nodes per position
--max-duration D Max duration per position (e.g. 5s)
--multipv N Number of PVs
--threads N Engine threads
--hash N Engine hash (MB)
--wdl / --no-wdl Toggle WDL output (if supported)

The engine flags do nothing on their own; they wake up only under --analyze, which runs a UCI engine to fold eval and PV motifs into the tag set before the model sees it. Leave --analyze off and the tags come straight from the static position — no engine, fully deterministic.

crtk fen text --fen "<FEN>" --model models/t5.bin --analyze -P config/default.engine.toml --max-duration 2s

Puzzle-line summaries: puzzle text

puzzle text takes a root puzzle FEN, walks out its principal variations, tags each step along the way, and summarizes the line as a whole. Here analysis is on by default — a puzzle without engine context is rarely worth narrating — so pass --no-analyze if you want to skip it.

Option Meaning
--model PATH T5 .bin model path (default: config t5-model-path)
--fen FEN Root puzzle FEN (required)
--multipv N Number of PVs to expand (default: 3)
--pv-plies N PV plies per line (default: 12)
--tag-multipv N Engine multipv during tag analysis (default: 1)
--max-new N Max generated tokens (default: 128)
--include-fen Emit JSON with fen + move + summary
--analyze / --no-analyze Enable (default) or skip per-move analysis
--protocol-path / -P PATH Engine protocol TOML file
--max-nodes N / --max-duration D Search limits per position
--threads N / --hash N Engine threads / hash (MB)
--wdl / --no-wdl Toggle WDL output (if supported) 
crtk puzzle text --fen "<FEN>" --model models/t5.bin --multipv 3 --pv-plies 12
crtk puzzle text --fen "<FEN>" --model models/t5.bin --include-fen --no-analyze

Its companion puzzle tags produces the same expanded positions as JSONL tag deltas and stops short of the text step — the cleanest way to see exactly what the summarizer is handed:

crtk puzzle tags --fen "<FEN>" --multipv 3 --pv-plies 12

The model file

Everything inference needs lives in one file. The runtime loads a custom binary export (extension .bin) that bundles:

  • A header magic LTTT and a format version (currently 1).
  • Model configuration: vocabulary size, model width, head dimension, FFN size, encoder/decoder layer counts, head count, relative-position bucket settings, and the special token ids (pad, EOS, decoder-start, unknown).
  • Flags for gated-GELU feed-forward and layer-norm epsilon.
  • The SentencePiece vocabulary: piece strings plus their unigram scores.
  • All weight tensors, keyed by export name (for example shared.weight for the shared embedding table), stored as little-endian float32.

This is crtk's own packaging format — not a Hugging Face checkpoint, not a .safetensors file. To use the feature you supply a .bin produced from a compatible T5-style encoder-decoder export. The repository bundles none.

Tokenizer

Encoding goes through a self-contained SentencePiece unigram tokenizer:

  • Input is whitespace-normalized, with spaces swapped for the SentencePiece ▁ marker.
  • A Viterbi search over the vocabulary picks the highest-scoring segmentation.
  • A span with no matching piece collapses to the unknown token, so encoding never fails — it degrades.
  • Decoding stitches the pieces back together and turns ▁ markers back into spaces.

The tokenizer rides inside the .bin, so encode and decode share one vocabulary with no external tokenizer files to keep in sync.

Prompt building

Before tokenizing, crtk wraps the tag list in a fixed instruction prompt. The instructions are pointed: write a single paragraph of chess commentary, ground it in material, king safety, and the top tactical motif, mention only moves that appear in the tags, and don't talk about tags or input formats. Then the tags follow as bullet lines, closed by an OUTPUT: marker.

With fen text and puzzle text you never touch this prompt — crtk derives the tags and assembles it. The shape matters when you train or fine-tune a model: the model has to have learned this tag-prompt format, or its output will drift.

Backends

Generation defaults to the CPU. When their JNI libraries are present, the native backends take over decoding; when one fails to load or stumbles at runtime, crtk drops back to the CPU path without complaint. A missing GPU library should never sink a run, so it doesn't.

Pick a backend with a JVM system property:

java -Dcrtk.t5.backend=auto -jar crtk.jar fen text --fen "<FEN>" --model models/t5.bin
Value Behavior
auto (default) Probe CUDA, then ROCm, then oneAPI; use the first available, else CPU
cpu Force the pure-Java CPU path
cuda Try the CUDA backend, fall back to CPU on failure
rocm (also amd, hip) Try the ROCm backend, fall back to CPU on failure
oneapi (also intel) Try the oneAPI backend, fall back to CPU on failure

These backends load the same way as crtk's other GPU paths (CUDA / ROCm / oneAPI), built from the native/ sources. Native GPU backends covers the build and detection details; crtk doctor or crtk engine gpu will tell you what your machine actually offers.

Low-level runner

The CLI is the supported entry point. Underneath it sits a thin standalone main that takes raw tags on stdin — the shortest path to the model when you want to debug it directly:

crtk fen tags --fen "<FEN>" | java -cp out chess.nn.t5.Main --model models/t5.bin
java -cp out chess.nn.t5.Main --model models/t5.bin --input tags.txt
Flag Meaning
--model PATH Path to the .bin export (required)
--input PATH Tag input file; reads stdin when omitted
--max-new N Max generated tokens (default: 128)
--debug Print token counts and top logits to stderr

It reads tags either as newline-separated lines or as a JSON array of strings — exactly what fen tags emits, so the two pipe together without translation.

Status and quality

The T5 path is experimental — built for research and tinkering, not as a finished commentary engine:

  • No weights ship with crtk. t5-model-path points at models/t5.bin, but that file is yours to provide from a compatible export.
  • Output is only as good as the model behind it. Load a weak or mismatched export and you'll get generic, repetitive, or off-topic text — the runtime is faithful to whatever it's given.
  • The runtime and .bin format are crtk's own simplified take, not a bit-exact reimplementation of any published checkpoint. Read the prose as a best-effort summary, never as ground truth.
  • When you need descriptions that are grounded and can't hallucinate, reach for position describe or the raw fen tags / puzzle tags output instead. T5 adds fluency on top of those same signals; it doesn't make them more true.

Determinism

Tag extraction and prompt building are deterministic, and CPU decoding is greedy — it always takes the highest-scoring next token — so a fixed model, prompt, and --max-new yield the same text every run. --analyze changes that calculus: it puts a UCI engine in the loop, and reproducibility then rests on that engine behaving deterministically under fixed node and time limits. The native backends try to match the CPU result but aren't guaranteed bit-identical. When you need the reference path, pin it with -Dcrtk.t5.backend=cpu.
