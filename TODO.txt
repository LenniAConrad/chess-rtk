### OTIS CUDA Backend

Goal: add an optimized CUDA implementation of the OTIS architecture and iterate
on it until it is as fast as practical while staying numerically correct.

- Implement the CUDA backend:
  - Add native CUDA kernels for OTIS inference, including input preparation,
    sheaf/Laplacian-style propagation, trunk blocks, policy logits, WDL/value
    output, and any required reductions.
  - Add JNI bindings and Java backend selection consistent with the existing
    NN CUDA/native backend patterns.
  - Support clean CPU fallback when CUDA is unavailable, model loading fails, or
    the current platform cannot load native libraries.
  - Keep memory ownership explicit: model upload, device buffers, batch-size
    assumptions, cleanup, and error reporting.
- Optimize through measured variants:
  - Start with a straightforward correct CUDA implementation.
  - Build and benchmark alternative kernels/layouts for the expensive stages:
    fused kernels, shared-memory tiling, coalesced access patterns, reduced
    host/device copies, persistent uploaded weights, and batched evaluation if
    it helps MCTS.
  - Compare CPU, naive CUDA, and optimized CUDA paths on the same positions and
    model files.
  - Pick the fastest implementation that preserves accuracy; do not trade away
    correctness for small speed gains.
  - Record benchmark hardware, driver/runtime version, model shape, batch size,
    latency, throughput, and numerical max/mean error.
- Accuracy and parity:
  - Add CPU-vs-CUDA parity tests for representative FENs.
  - Check policy logits, WDL/value outputs, intermediate tensors where exposed,
    and top-move ordering.
  - Define acceptable tolerances for float differences and fail loudly outside
    them.
  - Include edge cases: empty-ish boards, tactical positions, endgames,
    castling/en-passant rights, black-to-move orientation, and malformed model
    handling.
- Integrate with Workbench/CLI:
  - Add config/status reporting so Dashboard/Runtime diagnostics show OTIS CUDA
    availability, loaded backend, device name, and fallback reason.
  - Let OTIS-backed MCTS and Network views use the CUDA backend when selected
    and available.
  - Keep command-line evaluator/backend selection explicit and scriptable.
- Verification:
  - Compile native CUDA code and Java code from a clean checkout.
  - Run CPU-vs-CUDA parity tests.
  - Run OTIS MCTS smoke tests with CUDA enabled and disabled.
  - Benchmark several candidate implementations and keep the fastest accurate
    one, with results saved under `reports/` or documented in the wiki.
